2. the discovery pipeline (13 steps)

this is the full chain that wf-foundation-5x5 runs every time it fires. the rehearsal skill (mozart) replicates the same logic manually.

step 1. load foundation profile from d1
voice, icp, services, commission terms. this is the "brief" the agent carries for every run.
step 2. discover candidates (multi-source)
  • apify — linkedin profile search + jobs scraper for fractional cmo / advisor / consultant roles
  • exa.ai — semantic search for recent posts about reddit gtm, geo, youtube ads, linkedin growth
  • firecrawl — web scraping for personal sites, portfolio pages, podcast guest lists
step 3. dedupe against hubspot
checks existing contacts by email, linkedin url, and name+company match. prevents re-surfacing eli schwartz, kevin indig, etc.
step 4. enrich via browserbase + firecrawl
js-rendered pages (linkedin, substack) scraped for: full bio, client list, recent posts, audience size, service focus.
step 5. find + verify email (6-tier cascade)
find-email tool cascades through multiple sources. verifies via email list verify api. risk: some valid emails get dropped here — needs calibration.
step 6. score against ipp rubric v1.2
100-point rubric, 6 lenses. cheap model tier (haiku). outputs score + tier + rationale.
step 7. rank + pick top 5
mix of a-tier (75+) and b-tier (50-74). forces diversity, not just highest scores. c-tier (25-49) and dq (<25) are dropped.
step 8. write dossier (a/b only)
250-400 word structured brief per candidate. mid model tier (sonnet). includes portfolio fit, service signals, warm-intro path, outreach angle.
step 9. draft first-touch email
top model tier (sonnet). david's voice: hi {first}, 80-100 words, specific anchor, no links, "cheers, david" sign-off.
step 10. draft 4 follow-ups (tapered length)
cheap model tier (haiku). day 3 bump (<50 words), day 7 angle shift, day 12 value-add, day 18 close-the-loop (<30 words).
step 11. upsert hubspot contact + 11 custom properties
ipp_score, tier, rationale, partner_status, last_agent_run, partner_dossier, email_first_touch_body, followup_1_body..followup_4_body. risk: these custom props need to exist in hubspot before first write.
step 12. enroll in "foundation partner outreach" sequence (paused)
sequences are auto-generated but enrollment is paused. david reviews + unpauses the ones he wants to send.
step 13. log run to d1
run metadata, cost, token usage, candidates found, candidates written, errors. d1 table run_logs.

3. ipp scoring rubric (v1.2 — locked 2026-05-20)

lensweightwhat it measures
fit25title / role. tier 1 = 35 pts (fractional cmo, head of growth, marketing advisor, etc). tier 2 = 22 pts (director / vp going indie).
portfolio20client overlap with foundation icp. b2b saas, manufacturing, cybersecurity, education, b2b tech. $50m+ arr. us/canada.
network10ross 1st-degree + mutual connections. warm intro path.
audience20newsletter, podcast, linkedin following, creator velocity. 5k+ followers or active newsletter = strong signal.
intent20what they're posting. reddit content (most), youtube (high), geo/seo/social (medium). partner-platform mentions (profound, peec, etc) = bonus.
timing5recent role change, fresh budget, new practice launch.

tier mapping: a (75-100) → full treatment. b (50-74) → full treatment, lower priority. c (25-49) → watchlist, no drafts. dq (0-24 or zero icp overlap) → archived.

hard dq rule: if portfolio has zero icp overlap, auto-dq regardless of other scores.

6. what's working

  • multi-source discovery is robust. exa for semantic, apify for linkedin, firecrawl for web. three legs, less single-source-of-truth risk.
  • ipp rubric is specific. the 6-lens, 100-point system with hard dq prevents garbage-in. eli schwartz lesson is encoded.
  • rehearsal loop is the right iteration surface. david reviews, gives feedback, we tighten. no hubspot pollution during calibration.
  • d1 logging is comprehensive. every tool call, every run, every token. we can audit backward.
  • hubspot scopes are now fully wired. contacts + deals + associations + notes. pipeline stages mapped.
  • email voice is locked to david's spec. no links in cold emails, word ceilings enforced, "cheers, david" sign-off.

7. gaps, risks, and blind spots

risk 1. zero candidates written in all 3 runs so far
every run completed "successfully" but surfaced 0 candidates to hubspot. this means either: the scoring is too strict, discovery sources are returning junk, or the enrichment chain is failing silently. we cannot graduate to production until a run produces at least 3-5 viable candidates david agrees with.
risk 2. email verification cascade may be over-aggressive
if a candidate is perfect on every lens but email verify fails (email_disabled, invalid_mx), they get dropped. we need a fallback: flag for manual email lookup instead of hard-drop.
risk 3. hubspot custom properties may not exist yet
the 11 custom contact properties (ipp_score, tier, rationale, etc.) need to be created in hubspot before the workflow writes them. if they don't exist, the upsert will silently drop data or error.
risk 4. no dedupe against known bad leads
we dedupe against existing hubspot contacts, but we don't have a blocklist for candidates david has already reviewed and rejected. a rejected candidate could resurface on a future run if their hubspot record was deleted or if they have a different email.
risk 5. linkedin scraping brittleness
apify linkedin actors work well but linkedin rate-limits aggressively. if the actor is throttled, we may get partial profiles or empty results. no retry/backoff logic is visible in the current pipeline.
risk 6. no signal for when a candidate's situation changes
once a c-tier candidate is watchlisted, we don't monitor them. if they post a viral reddit thread next month, get a podcast, or go full-time fractional, we miss it. no re-evaluation trigger.
risk 7. sequence is hubspot-native, not instantly
if hubspot sequences have volume limits or deliverability issues, we have no fallback. instantly is in the stack but not wired as a fallback in the workflow.

8. recommended improvements

8.1. fix the zero-candidates problem (priority 1)

  • run 3-5 rehearsals this week with david reviewing each.
  • adjust scoring thresholds: consider "b-tier with strong intent" as valid candidates even if portfolio is thin.
  • add a manual override: if david gives a name, bypass scoring and run enrichment directly.
  • log the full discovery output (even dq'd candidates) so we can see where the funnel drops.

8.2. add a "soft pass" for email failures (priority 2)

  • if email verify fails but candidate is a-tier, create the hubspot contact anyway with a "needs email" flag.
  • david or mozart can manually source the email via rocketreach, zoominfo, or linkedin dm.

8.3. audit hubspot custom properties (priority 2)

  • confirm all 11 properties exist in hubspot. if any are missing, create them before first production write.
  • properties: ipp_score (number), tier (dropdown: a/b/c/dq), rationale (textarea), partner_status (dropdown: sourced/engaged/agreement/activated/first referral/revenue/nurturing), last_agent_run (date), partner_dossier (textarea), email_first_touch_body (textarea), followup_1..4_body (textarea each).

8.4. build a rejection blocklist (priority 3)

  • when david rejects a candidate, log to d1 rejections table with reason.
  • future runs check this table before scoring. prevents resurfacing.
  • optionally add an "auto-retry in 6 months" flag for timing-dependent rejections.

8.5. add retry/backoff for linkedin scraping (priority 3)

  • if apify returns empty, retry once after 30s with a different actor or proxy.
  • fallback: use exa to find their latest post as a signal even if linkedin profile is blocked.

8.6. lightweight signal monitoring for watchlisted contacts (priority 4)

  • once a week, check c-tier candidates for: new linkedin post, new newsletter issue, role change, podcast appearance.
  • if signal strength increases, bump to b-tier and alert david.
  • this prevents us from missing a candidate who "grows into" the icp.

8.7. wire instantly as a fallback sender (priority 4)

  • if hubspot sequence enrollment fails or rate-limits, log to d1 and optionally trigger an instantly campaign.
  • this is mostly insurance — only needed if hubspot becomes a bottleneck.

8.8. manual lead ingestion workflow (new capability, priority 2)

david will hand us leads in 3 shapes:

  • csv bulk list — linkedin urls, names + companies, or raw exports. we parse, enrich, score, draft, push to hubspot in batch.
  • one-at-a-time — david texts a name or linkedin url. we run the full pipeline for that single contact, return a dossier + draft in ~2 minutes, ask if he wants it in hubspot.
  • handful (3-10) — similar to csv but smaller. we can run these synchronously in one rehearsal-style page for review.

this should be a distinct mozart skill: foundation/ingest-leads. same pipeline as 5x5 but with csv/linkedin/name input instead of agent discovery.

  3. c-tier monitoring: do you want to be notified when a watchlisted candidate posts something relevant, or should that be silent until they cross into b-tier?
  4. manual lead ingestion: what's your most common input format? linkedin url, name + company, or csv export? we should optimize the happy path.
